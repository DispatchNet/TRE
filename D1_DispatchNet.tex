\section*{Purpose}
This document describes the scope of the Project "TRE", which implements a system for rail transport management.

The purpose of this document is to: 
\begin{itemize}
	\item Present the objectives.
	\item Identify the primary stakeholders.
	\item Define the functional and non-functional requirements.
\end{itemize}

\section{Objectives}
The project provides an interface that covers the needs of both \un{passengers} and train companies, as well as an underlying 
management system of rolling stock and network. 
The solution to this project has been identified in an application accessible for the aforesaid users.

Specifically, the objectives of the project are:

\subs{Ticketing system}{O.1}
Subsystem which deals with tickets, from the search to the possible actions after the purchase. 
It includes basic operations that the user can perform, such as (but not limiting to):
\begin{itemize}
	\item Selection with details.
	\item Purchase.
	\item Previous purchase history.
	\item Cancellation
\end{itemize}

\it{Boundary:} The project assumes the use of a third-party payment system, implementing only a place-holder for a real 
integration with it.

\subs{Route-finding}{O.2}
The system lets users select a starting station, a destination station, and either a departure time or expected arrival time.
The system then automatically computes and ranks the possible \bf{routes} that best fit the user's request and permissions
(for example, not permitting cargo trains to \un{Passengers}).
The user must be able to choose among the options (if multiple), comparing estimated arrival and departure times of each 
train and possible count and locations of transfers.
The system includes a dedicated section which displays several rankings based on the user's preferences.

\subs{Virtual board}{O.3}
The system provides users any station's Departure and Arrival boards, and, similarly, all stations in route for 
individual train lines, including all the expected scheduled times for each.

\subs{Rail system modeling}{O.4}
The system provides a data-driven approach to rail network modelling, such that it can be efficiently traversed by trains, 
and therefore goods and \un{passengers}.

\subsubsection*{Human}
\un{Network managers}, the product owner. They want to combine the availability of train organizations with \un{passengers'} needs.
They are the primary stakeholder for making functional decisions.

\subsection*{External}

\addcontentsline{toc}{subsubsection}{Human}
\subsubsection*{Human}
\un{Passengers}, the main end users. They want to streamline their interactions with the train transport system, including:
\begin{itemize}
	\item Searching for routes and schedules.
	\item Analyzing departure/arrival boards as well as train information.
	\item Buying tickets.
\end{itemize}

\un{Anonymous Users}, which are \un{passengers}, save for not having logged in to an account, thus not 
having access to personal and personalized system functionalities.
Upon logging in, \un{Anonymous Users} may prove to not be \un{Passengers}, but are assumed to be \un{passengers} until proved 
otherwise. \\

\noindent\un{Train companies}, who provide the rolling stock. They want an efficient management of the network 
infrastructure. They are also expected to provide information of trains they ask to be managed.

\section{Functional requirements}
\subs{Account classification}{FR.1}
The system shall classify between \un{anonymous user}, \un{passenger}, \un{train company} operator and \un{network manager}. 
The additional functionalities are described in 
\hyperref[Table1]{\bf{Requirement restrictions based on user classification}}.

\subs{Account registration}{FR.2}
Following \hyl{FR.1}, in reference to \hyl{NFR.2}, the system must allow \un{anonymous users} to register. Data must 
satisfy \hyl{NFR.3}. Password must satisfy \hyl{NFR.4} and \hyl{NFR.5}.

\subs{Consent management}{FR.3}
Following \hyl{NFR.6}, in reference to \hyl{FR.2}, at registration the system must request the user's consent for personal 
data processing.

\subs{Account registration notification}{FR.4}
In reference to \hyl{FR.2}, after successful registration the system must notify the user as described at \hyl{NFR.9}.

\subs{Account deletion}{FR.5}
Following \hyl{NFR.1}, the system shall allow users to delete their accounts, as well as any other related data.

\subs{Account deletion notification}{FR.6}
In reference to \hyl{FR.5}, after successful deletion the user is notified as described at \hyl{NFR.9}.

\subs{Data request management}{FR.7}
Following \hyl{NFR.7}, in reference to \hyl{NFR.2}, the system must provide users view of their stored information.

\subs{Account data change}{FR.8}
Following \hyl{NFR.8}, in reference to \hyl{NFR.2}, the system must allow users to change their data. New data must 
satisfy \hyl{NFR.3}.

\subs{Account data change notification}{FR.9}
In reference to \hyl{FR.8}, after a successful change operation the user is notified as described at \hyl{NFR.9}.

\subs{Password reset}{FR.10}
Following \hyl{NFR.8}, in reference to \hyl{NFR.2}, the system shall allow users to reset the password and change it. 
New password must satisfy \hyl{NFR.4} and \hyl{NFR.5}.

\subs{Login}{FR.11}
Following \hyl{FR.1}, the system must let users login using their username and password defined at registration (\hyl{FR.2}).

\un{Boundary:} The \un{network manager} has default account username "admin" and password "Password1234".

\subs{Logout}{FR.12}
Following \hyl{FR.1}, the system shall let logged in users log out, switching to the restricted \un{anonymous user's} 
functionalities. 

\subs{Tickets details display}{FR.13}
Following \hyl{O.1}, the system must allow users to view tickets. 

\subs{Tickets purchase}{FR.14}
Following \hyl{O.1}, the system must allow users to buy tickets.

\subs{Tickets purchase confirmation}{FR.15}
Following \hyl{O.1}, in reference to \hyl{FR.14}, the user must be asked for a final confirmation before purchase.

\subs{Tickets purchase notification}{FR.16}
Following \hyl{O.1}, in reference to \hyl{FR.14}, at the end of a purchase operation, the system must notify the user as 
described at \hyl{NFR.9}.

\subs{Tickets cancellation}{FR.17}
Following \hyl{O.1}, the system must allow the cancellation of purchased tickets. 

\subs{Tickets cancellation confirmation}{FR.18}
Following \hyl{O.1}, in reference to \hyl{FR.17} and \hyl{NFR.9}, the user must be asked for a final confirmation before 
cancellation.

\subs{Tickets cancellation notification}{FR.19}
Following \hyl{O.1}, in reference to \hyl{FR.17}, at the end of a cancellation operation, the system must notify the user as 
described at \hyl{NFR.9}.

\subs{Tickets history}{FR.20}
Following \hyl{O.1}, the system must allow users to view their tickets history, both purchased and cancelled.

\subs{Tickets classification}{FR.21}
Following \hyl{O.1}, in reference to \hyl{FR.20}, the tickets history must show whether a ticket is valid, expired, or 
cancelled.

\subs{Path search}{FR.22}
Following \hyl{O.2}, the system needs to be able to compute multiple paths between stations.

\subs{Path points}{FR.23}
Following \hyl{O.2}, extending \hyl{FR.22}, the user must choose both the first and last stations of paths to be computed.

\subs{Path timing}{FR.24}
Following \hyl{O.2}, extending \hyl{FR.22}, the user can, optionally, specify a first departure time and/or a last 
arrival time. If neither is specified, the system must default to the earliest departure from the current time on-wards.

\subs{Path ranking}{FR.25}
Following \hyl{O.2}, in reference to \hyl{FR.22}, the system needs to be able to rank the paths over the network by one of
several factors, at user's choice among a fixed list.

\subs{Ranking length}{FR.26}
Following \hyl{O.2}, in reference to \hyl{FR.25}, the system shall be able to rank higher the paths with the least 
travel time, and the ones with the least amount of stations 
among them.

\subs{Ranking deviation "Departure"}{FR.27}
Following \hyl{O.2}, in reference to \hyl{FR.25}, the system shall be able to rank higher paths that have a first 
departure time closest (but not earlier than) the user requested departure time.

\subs{Ranking deviation 2 "Arrival"}{FR.28}
Following \hyl{O.2}, in reference to \hyl{FR.25}, the system shall be able to rank higher the paths that have a last 
arrival time closest (but no later than) the user requested departure time.

\subs{Ranking cost}{FR.29}
Following \hyl{O.2}, in reference to \hyl{FR.25}, the system shall be able to rank higher the paths with lower total cost of 
tickets' not already owned as valid by the user.


\subs{Transfers}{FR.30}
Following \hyl{O.2}, in reference to \hyl{FR.22}, the system needs to be able to compute transfer paths out of 
intersecting routes.

\subs{Segregation}{FR.31}
Following \hyl{O.2}, in reference to \hyl{FR.22}, the system must not consider train routes that pass by trains that the user is not allowed to purchase a ticket for.

\subs{Station display}{FR.32}
Following \hyl{O.3}, the system must provide departure and arrival times of all trains that pass through any station.

\subs{Train display}{FR.33}
Following \hyl{O.3}, the system must provide departure and arrival times of a train at all the stations on its schedule.

\subs{Path display}{FR.34}
The system must be able to provide the board described at \hyl{FR.33} for all trains in a path, within the bounds of the train's participation in 
that path.

\subs{Creating a Service Request}{FR.35}
Following \hyl{O.4}, \un{Companies} must be allowed to send the system a request for the introduction of a service.

\subs{Defining a Train Type}{FR.36}
Following \hyl{O.4}, in reference to  \hyl{FR.37}, the system must allow companies to register a train type to be used when defining a service.

\subs{Defining a Service Type}{FR.37}
Following \hyl{O.4}, in reference to \hyl{FR.35} and \hyl{FR.36}.  
The system must allow companies to define services and classes, as well as a pricing model for passenger trains.

\subs{Viewing the Network}{FR.38}
Following \hyl{O.3} and \hyl{O.4}, the system must allow users to access a map of the network.

\subs{Registering a Station}{FR.39}
Following \hyl{O.4}, the system must allow \un{network managers} to register a station.

\subs{Modifying a Station}{FR.40}
Following \hyl{O.4}, the system must allow \un{network managers} to modify any existing station.


\subs{Deleting a Station}{FR.41}
Following \hyl{O.4}, the system must allow \un{network managers} to delete any existing station.

\subs{Registering a Line}{FR.42}
Following \hyl{O.4}, the system must allow \un{network managers} to register a line.  


\subs{Modifying a Line}{FR.43}
Following \hyl{O.4}, the system must allow \un{network managers} to modify any existing line.  

\subs{Deleting a Line}{FR.44}
Following \hyl{O.4}, the system must allow \un{network managers} to delete any existing line.

\subs{Registering a Junction}{FR.45}
Following \hyl{O.4}, the system must allow \un{network managers} to register a junction.  

\subs{Modifying a Junction}{FR.46}
Following \hyl{O.4}, the system must allow \un{network managers} to modify any existing junction.  

\subs{Deleting a Junction}{FR.47}
Following \hyl{O.4}, the system must allow \un{network managers} to delete any existing junction.  

\subs{Automated Service Integrity Checks}{FR.48}
Following \hyl{O.4}, in reference to \hyl{FR.55}, the system must be able to autonomously perform integrity checks on proposed services.
Integrity checks are defined by: \hyl{FR.50}, \hyl{FR.51}, \hyl{FR.52}, and \hyl{FR.53}.

\subs{Displaying Integrity Check Results}{FR.49}
Following \hyl{O.4}, extending \hyl{FR.48}, the system must be able to provide these results.

\subs{Single Track Line Availability Check}{FR.50}
Following \hyl{O.4}, specifying \hyl{FR.48}, the system must ensure that no two services are scheduled to traverse a single track line in the 
opposite directions at the same time.

\subs{Platform Availability Check}{FR.51}
Following \hyl{O.4}, specifying \hyl{FR.48}, the system must ensure that no two services are scheduled to stop at the same platform at the same time.

\subs{Track-side Systems Compatibility Check}{FR.52}
Following \hyl{O.4}, specifying \hyl{FR.48}, the system must ensure that all trains used for a service are compatible with the signaling and 
electrification systems installed on the traversed infrastructure.

\subs{Track Geometry Check}{FR.53}
Following \hyl{O.4}, specifying \hyl{FR.48}, the system must ensure that all trains used for a service are compatible with both the loading 
and track gauges of the infrastructure they traverse.

\subs{Service Request Review}{FR.54}
Following \hyl{O.4}, in reference to \hyl{FR.35}, the system must allow the \un{network managers} to review requests.

\subs{Service Request approval}{FR.55}
Following \hyl{O.4}, in reference to \hyl{FR.35} and \hyl{FR.54}, the \un{network manager} must be able to approve a service request, applying the proposed changes to 
the system and notifying (\hyl{NFR.9}) the \un{train company} that submitted the request.

\subs{Service Request amendment}{FR.56}
Following \hyl{O.4}, in reference to \hyl{FR.35} and \hyl{FR.54}, the system must allow the \un{network manager} to send back an amended version of the request for 
the \un{train company} to review, notifying (\hyl{NFR.9}) them.

\subs{Service Request denial}{FR.57}
Following \hyl{O.4}, in reference to \hyl{FR.35} and \hyl{FR.54}, \un{network managers} must be able to deny a service request, notifying (\hyl{NFR.9}) the \un{train company} that submitted the request.

\section{Non-functional requirements}

\subs{Data Protection Regulation}{NFR.1}
\un{External requirement - Legislative requirement} \\

\noindent The system respects the \href{https://eur-lex.europa.eu/legal-content/EN/TXT/?uri=CELEX%3A02016R0679-20160504&qid=1532348683434}
{Regulation (EU) 2016/679} on the protection of natural persons with regard to the processing of personal data. In particular:
\begin{itemize}
	\item[Art. 5] Principles relating to processing of personal data, ensuring ethical processing.
	\item[Art. 6, 7] Lawfulness of processing and Conditions for consent, both concerning consent for the processing 
	of personal data.
	\item[Art. 13] Information to be provided where personal data are collected from the data subject, 
	i.e. providing information of the controller of the processing.
	\item[Art. 15] Right of access by the data subject, i.e. users must be able to see their stored information.
	\item[Art. 16] Right to rectification, i.e. users must be able to change their information.
	\item[Art. 17] Right to erasure (‘right to be forgotten’).
	\item[Art. 32] Security of processing.
\end{itemize}

\subs{Account data}{NFR.2}
\un{Product requirement - Security requirement} \\

\noindent Following \hyl{FR.1} and \hyl{NFR.1}, in reference to \hyl{FR.2}, user accounts must include username, email and 
password.

\subs{Account data details}{NFR.3}
\un{Product requirement - Security requirement} \\

\noindent Following \hyl{NFR.1} and \hyl{NFR2}, account data must satisfy the following:
\begin{itemize}
	\item The username must be unique.
	\item The email must be unique.
\end{itemize}

\subs{Password security}{NFR.4}
\un{Product requirement - Security requirement} \\

\noindent Following \hyl{NFR.1} and \hyl{NFR.2}, in reference to \hyl{FR.3}, the password must be at least 8 characters, 
with at least one upper and one lowercase letters and one digit.

\subs{Typo prevention}{NFR.5}
\un{Product requirement - Security requirement} \\

\noindent When a password is chosen or modified, the system must take prevention to mitigate typos: it must require the password to be typed twice and only accepted if both match.

\subs{Consent for personal data processing}{NFR.6}
\un{External requirement - Legislative requirement} \\

\noindent Following \hyl{NFR.1}, the system must comply with the GDPR for data processing consent.

\subs{Request of processed data}{NFR.7}
\un{External requirement - Legislative requirement} \\

\noindent Following \hyl{NFR1}, the system must comply with the GDPR for accessing processed data.

\subs{Rectification of processed data}{NFR.8}
\un{External requirement - Legislative requirement} \\

\noindent Following \hyl{NFR1}, the system must comply with the GDPR for the right of rectification.

\subs{Email notification}{NFR.9}
\un{Product requirement - Usability requirement} \\

\noindent Notifications shall be sent via email to the account's email address.

\subs{Station to train display}{NFR.10}
\un{Product requirement - Usability requirement} \\

\noindent Users must be able to switch with a single prompt/input between a station's display and a train's display passing 
through that station.

\subs{Train to station display}{NFR.11}
\un{Product requirement - Usability requirement} \\

\noindent Users must be able to switch with a single prompt/input between a train's display and a station's display in its 
schedule.

\subs{Station "from" path}{NFR.12}
\un{Product requirement - Usability requirement} \\

\noindent Users must be able to switch with a single prompt/input between a station's display and the route-finding 
procedure with the station automatically set as the path's start.

\subs{Station "to" path}{NFR.13}
\un{Product requirement - Usability Requirement} \\

\noindent Users must be able to switch with a single prompt/input between a station's display and the route-finding 
procedure with the station automatically set as the destination.

\subs{Path list to path list}{NFR.14}
\un{Product requirement - Usability requirement} \\

\noindent Users must be able to switch from a list of paths according to a ranking option to another, in no more 
steps/inputs than the ones required to select a different ranking. 

\subs{Ranking non-choice}{NFR.15}
\un{Product requirement - Usability requirement} \\

\noindent The system shall default to the ranking described at \hyl{FR.26} until a different ranking choice is given by the user.
